## The Problem: Manual Configuration

| Problem | Description |
|---|---|
| **Server Drift** | Servers silently diverge from each other ‚Äî one has a config tweak, another doesn't |
| **Snowflake Servers** | Every server is unique and irreplaceable ‚Äî delete it and the knowledge is gone |
| **No Audit Trail** | Manual changes leave no trace and no rollback path |
| **Doesn't Scale** | 3 servers: manageable. 300: impossible without automation |

**The Goal:** Declare the desired state ‚Üí let automation handle the rest  
‚úì Every server identical  
‚úì Changes tracked in Git  
‚úì Rollback in seconds  
‚úì Scale to thousands

---

## What is Configuration Management?

| Concept | Description |
|---|---|
| **Desired State Declaration** | Describe WHAT you want; the tool figures out HOW to get there |
| **Idempotency** | Running the same playbook 10 times = running it once ‚Äî only changes what needs changing |
| **Version Control** | Config lives in Git ‚Äî every change has author, timestamp, message, rollback path |
| **Consistency & Auditability** | Every server matches the declared spec; compliance teams can audit the Git history |

## Ansible Architecture

```
CONTROL NODE (ansible-control)        MANAGED NODES
  Ansible installed                   node1:  SSH + Python (no Ansible)
  Playbooks live here         SSH ‚Üí   node2:  SSH + Python (no Ansible)
  inventory.ini               SSH ‚Üí
  ansible.cfg
  SSH private key

KEY POINT: Ansible is only installed on the CONTROL NODE.
```

---

## How Ansible Works: Step by Step

1. **You run** `ansible-playbook` on the control node
2. **Ansible reads the inventory** ‚Äî finds all hosts in target groups
3. **Opens SSH connections** ‚Äî parallel sessions to every target host
4. **Uploads Python modules** ‚Äî copies tiny Python scripts to `/tmp` on each node
5. **Modules report back** ‚Äî each returns `changed` / `ok` / `failed`
6. **SSH connections close** ‚Äî no persistent agent running

> All steps 3‚Äì6 happen **in parallel** across all hosts simultaneously

## Key Ansible Concepts

| Concept | Description |
|---|---|
| **Inventory** | List of hosts and how to connect; groups let you target subsets `[web]`, `[db]` |
| **Playbook** | Ordered YAML file of plays ‚Äî top-level file passed to `ansible-playbook` |
| **Play** | Maps a group of hosts (`hosts: web`) to a list of tasks |
| **Task** | One action ‚Äî calls a module with parameters; reports ok/changed/failed |
| **Module** | Built-in unit of work: `apt`, `copy`, `service`, `template`, `file`, `command` (6,000+) |
| **Handler** | Task that runs ONLY when notified ‚Äî e.g., restart nginx only if config changed |
| **Role** | Reusable, structured collection of tasks, templates, variables, handlers |
| **Variable** | Parameterize playbooks via `host_vars`, `group_vars`, vars files, or `--extra-vars` |

## File Deep Dives

### `Dockerfile` (Managed Nodes)
```dockerfile
FROM ubuntu:22.04
RUN apt install openssh-server python3  # Ansible connects via SSH + runs Python modules
RUN useradd ubuntu && chpasswd          # SSH user Ansible will log in as
RUN adduser ubuntu sudo + NOPASSWD:ALL  # Enables become: true without password prompt
CMD ["/usr/sbin/sshd", "-D"]           # Keeps container alive with SSH server
```

### `Dockerfile.control` (Control Node)
```dockerfile
FROM ubuntu:22.04
RUN apt install openssh-client sshpass  # SSH client + key distribution
RUN pip3 install ansible                # Only here ‚Äî NOT on managed nodes
COPY entrypoint.sh + ENTRYPOINT         # Bootstrap runs on startup
```

### `entrypoint.sh`
1. Create `/root/.ssh` with correct permissions
2. Generate RSA key pair (`ssh-keygen`)
3. Copy public key to `node1` and `node2` via `ssh-copy-id` (using sshpass for initial password auth)
4. Retry loop ‚Äî wait until nodes are ready before copying
5. `exec tail -f /dev/null` ‚Äî keep container running as daemon

### `ansible.cfg`
```ini
[defaults]
inventory = inventory.ini
host_key_checking = False    # Safe for lab; keep True in production
retry_files_enabled = False  # Clean directory during demos
```

### `inventory.ini`
```ini
[web]
node1
node2

[web:vars]
ansible_user=ubuntu
ansible_ssh_private_key_file=/root/.ssh/id_rsa
```

## Playbook Examples

### `install_nginx.yml` ‚Äî Playbook 1 (minimal)
```yaml
- name: Install nginx
  hosts: web
  become: true
  tasks:
    - name: Install nginx package
      apt:
        name: nginx
        state: present      # IDEMPOTENT: install if missing, skip if already there
        update_cache: yes
```

### `deploy_website.yml` ‚Äî Playbook 2 (multi-task)
```yaml
- name: Deploy website
  hosts: web
  become: true
  tasks:
    - name: Install nginx
      apt:
        name: nginx
        state: present

    - name: Create website
      copy:
        dest: /var/www/html/index.html
        content: |
          <html><body>Hello from Ansible</body></html>

    - name: Start nginx
      service:
        name: nginx
        state: started
```

### `templates/nginx.conf.j2` ‚Äî Jinja2 Template
```
server {
    listen {{ nginx_port }};
    location / {
        return 200 'Hello from Ansible';
    }
}
```
Same template, different environments:
- Production: `port=80`, Staging: `port=8080`, Dev: `port=3000`

## Reading Ansible Output

```
PLAY RECAP *****************************
node1  : ok=4  changed=2  unreachable=0  failed=0
node2  : ok=4  changed=2  unreachable=0  failed=0
```

| Field | Meaning |
|---|---|
| `ok` | Task ran ‚Äî state was already correct (idempotency working) |
| `changed` | Task ran and made a real change |
| `unreachable` | Ansible couldn't SSH into the host |
| `failed` | Connected but task reported an error |

> **Idempotency test:** Run the same playbook twice ‚Üí second run shows `changed=0`

## Docker Networking ‚Äî Why Containers Talk by Name

```yaml
networks:
  ansible-net:
    driver: bridge

# Each service attaches to:
    networks:
      - ansible-net
```

- Docker's embedded DNS resolves container names to IPs automatically
- `node1` ‚Üí resolves to its container IP
- Without explicit named network: hostname resolution may break, Ansible ping fails

## Key Takeaways

1. **Agentless**: Ansible needs only SSH + Python on managed nodes ‚Äî no agent installation
2. **Idempotency**: Running a playbook 10 times = running it once ‚Äî safe for production
3. **Playbook = WHAT | Inventory = WHO | ansible.cfg = HOW**
4. `become: true` = sudo; `state: present` = idempotent install
5. Docker networking with named bridges enables DNS-based hostname resolution
6. `entrypoint.sh` distributes SSH keys on startup ‚Äî foundation of passwordless auth
7. Roles make playbooks reusable and testable ‚Äî use them for production systems
